Building Tools That Actually Help Your Team

Lessons from shipping real tools — why starting small, solving your own pain, and iterating fast beats big planning.

I've built several tools now — some that people use daily, and some that quietly faded away. The ones that stuck have a few things in common.

Not technical brilliance. Not elegant code. They stuck because they solved a real, specific, recurring pain point for someone. Here's what I've learned about building tools that actually help.

Solve Your Own Problem First

Every tool I've built started with a frustration I had myself. Precision Match came from spending hours comparing resumes to job descriptions. Noob → AI Expert came from wanting a structured path to teach AI without the fluff. Prompt of the Day came from wanting to practice AI prompting consistently.

When you build for yourself, you have a built-in product manager — you. You know exactly what matters, what's annoying, and what can wait. You don't need user research to figure out the requirements. Your own frustration is your compass.

Start Embarrassingly Small

The first version of Precision Match was a dozen lines of Python. It didn't have a UI. It didn't handle edge cases. But it did give me a useful result in 30 seconds instead of 30 minutes.

That's the bar for version one: does it save time on a task you'd otherwise do manually? If yes, ship it. Add polish later. Most tools never get built because people overestimate what v1 needs to be and underestimate what v2 can become.

Optimize for the 80% Use Case

Early in my building journey, I spent weeks trying to handle every possible input format, error state, and edge case. The result? A robust tool that nobody used because I'd run out of steam before making it accessible.

Now I follow the 80/20 rule: build for the 80% use case that covers most users, and handle the edge cases when they actually come up. Most of the time, they never do. And when they do, you have real examples to test against — not hypothetical ones.

Ship, Then Ask

I used to ask teams what tools they needed. The answer was always vague: "something to help with reporting" or "automation would be nice." Nobody can specify a tool they've never seen.

Now I ship something small, show it to one person, and ask: "Does this help? What would make it better?" Concrete feedback on a real thing is infinitely more valuable than abstract requirements gathering.

Tools Don't Need to Be Perfect

Some of my most-used tools have rough edges. The design isn't polished. The code isn't clean. But they solve a problem, and that's enough. A tool that's 80% useful and shipped today is better than a 100% tool that ships next quarter — or never.

If you're in IT and thinking about building something, my advice is simple: pick one frustration, build the simplest thing that could possibly work, and put it in front of someone today. Not next week. Today.
